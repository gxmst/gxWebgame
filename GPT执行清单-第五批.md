## 背景（先读懂当前实现再改）

关于阶段 2，根因已定位，改的时候要对症，不要瞎调：

- 镜头缩放在 `camera.js` 的 `#refreshMassZoom()`：`targetZoom = 1 / (displayMass/mass.start)^massZoomExponent`（`massZoomExponent=0.2`），结果被 `clamp(minZoom, maxZoom)` 夹死，`minZoom=0.48`、`maxZoom=1`。
- **问题**：玩家越大镜头本应拉越远，但拉到 `minZoom=0.48` 就触底不再拉远 → 玩家和鱼在屏幕上越来越大 → 后期"布满屏幕"。
- **同一个问题的另一面**：水雷/渔网是固定世界尺寸（水雷半径约 23），镜头触底后玩家显得巨大、危险物显得像芝麻甚至看不见 → 封神后被看不见的水雷炸死。
- 渔网现有实现：`createNetWarning` + `updateSpecials` 里 `net` 分支，是一道从上方下落的网，目前**只判定玩家**（`resolveCollisions` 里 net 只和 playerBody 比），碰到其他鱼没有任何效果。

## 阶段 1：程序化舒缓电子背景音乐

**目标**：加一段**不吵、舒缓、氛围向**的电子背景音乐，用 Web Audio API **程序化实时合成**（不引入任何音频文件）。可在设置界面开关、受总音量控制。

**做法**：

1. 在现有 `audio.js`（AudioSystem）基础上，新增一个**背景音乐层**。用 Web Audio 的振荡器 / 增益 / 简单滤波，合成缓慢、柔和的电子氛围音（pad / drone 风格）：低音量、长音符、缓慢起伏，**避免尖锐、密集、打拍子的吵闹音**。可以用几个和谐的音（如一个小调和弦的分解 / 缓慢琶音）循环，配合缓慢的音量呼吸，营造深海氛围。
2. **接入现有设置**：
   - 背景音乐音量受现有总音量 `save.settings.volume` 和静音 `save.settings.muted` 控制。
   - 新增一个独立的**"背景音乐"开关**（设置界面加一个 toggle，`save.settings.music` 之类，默认开或关你定，建议默认开但音量偏低）。关掉只停音乐，不影响音效。
3. **浏览器自动播放限制**：页面加载不能自动出声，必须用户交互后才能播。复用现有的 `audio.unlock()`（现在点击已经在 unlock）——音乐在玩家第一次交互（点开始/任意点击）后才启动。
4. **状态适配**：游戏进行中播放；暂停 / 标题页可继续轻声播放或降低音量（你定，保持舒缓不突兀）。切后台（`visibilitychange` / `blur`）时应停止或静音，回来恢复，避免后台还在响。
5. 所有音乐参数（音量、节奏快慢、音高、和弦等）**放进 `config.js` 一个 `music` 段**，带注释可调。

**阶段 1 自检**：进游戏后能听到舒缓背景音乐；设置里关掉音乐后音乐停、音效仍在；调总音量音乐跟着变；切后台不再响；控制台无 Web Audio 报错。

## 阶段 2：后期世界比例修正（镜头 + 体型软封顶 + 危险物可见）

**目标**：解决"后期鱼布满屏幕"和"封神后被看不见的水雷炸死"。三个子项配合成一套自洽的比例规则。

### 2.1 成长期（T1~T6）：镜头能拉更远
- 把 `camera.minZoom` 放低（例如 0.48 → 0.30 左右，具体值进 config 可调），让玩家变大时镜头能继续拉远，缓解"还没封神就开始糊屏"。
- 保持 `massZoomExponent` 的平滑缩放手感，只是把下限放宽，不要改成突变。

### 2.2 封神后：体型软封顶
- 玩家进入自由霸主模式（sovereign / ENDLESS）后，**体型基本不再显著增长**（或增长极缓、趋于一个上限）。封神后玩法是巡游刷分，体型大小已无意义，软封顶后：
  - 镜头缩放随之**稳定在一个"能看清全局"的远景**，不再触底失控。
  - 玩家不再无限膨胀布满屏幕。
- 软封顶的体型上限值、封顶后是否允许极缓增长，都进 `config.js` 可调。吃鱼仍照常加分（封顶的是"体型/视觉"，不是"分数"）。

### 2.3 致命危险物保证最小可见尺寸
- 渔网、水雷这类**会直接致死**的环境威胁，在**渲染和判定上都保证一个"最小屏幕可见尺寸"**：不管镜头拉多远、玩家多大，它们都不会小到看不见或看不清。
- 具体做法：绘制时若按世界尺寸×zoom 算出的屏幕半径小于某个下限（进 config，如"最小屏幕像素半径"），则按下限画，并给它们保留清晰的警示标记（现有渔网有黄色警示、水雷有引信标记，确保这些在远景下依然醒目）。
- **注意判定与视觉的一致性**：如果为了可见把它放大显示，要让玩家的"看到的危险范围"和"实际致死判定范围"大致对应，避免"看着没碰到却死了"的憋屈感。宁可判定略宽容（危险物判定半径比视觉略小一点点），也不要让玩家死得莫名其妙。

**阶段 2 自检**：
- 从 T1 玩到 T6，全程玩家和鱼在屏幕上的大小合理，不会糊满屏。
- 封神后体型稳定、镜头稳定在全局远景。
- 封神后（镜头很远时）渔网和水雷依然清晰可见、有醒目警示，能提前看到并躲开，不会"看不见就被炸死"。

## 阶段 3：渔网路径清鱼（封神巡游的核心威胁）

**目标**：让渔网不只威胁玩家，也**把它下落路径扫过的鱼一起捕走**，成为"移动的清场威胁"，逼玩家在封神巡游时不断转移、追着鱼群跑、躲着渔网走。**力度要克制——不是一按清空全屏，而是只清渔网那一条竖直下落带里的鱼。**

**做法**：

1. 渔网下落（`net` active 阶段）时，除了判定玩家，也判定**所有鱼**：落入渔网横向宽度范围内、且被网扫到的鱼，**被捕获（`active=false`）并消失**。可加一个轻量的被捕特效（挣扎小动效 / 少量气泡），但**务必轻量**（一次可能捕多条，粒子要省，遵守现有粒子预算）。
2. **范围限定**：只影响渔网**横向宽度 × 下落路径**这一条竖直带，不是全屏。带外的鱼不受影响。
3. **清场后补鱼**：渔网清走一批鱼后，生成器（director）要在**别处（渔网带外、玩家视野内外合理位置）逐步补回鱼**，维持世界鱼群总量，避免整个世界越玩越空。补鱼节奏平滑，别在玩家眼皮底下凭空冒出。
4. **玩法咬合**：这个机制主要服务封神后的自由巡游——渔网既致死（碰到玩家=死，已有）、又抢食（把鱼捕走），玩家得避开渔网清过的空区、往鱼还多的地方游。成长期（T1~T6）渔网出现较少，此机制影响有限；封神后渔网随时间变密（现有 `sovereignHazards` 逻辑），清场威胁逐步增强，但保持"可生存"、有节奏，别让玩家封神后马上无鱼可吃或被秒。
5. 相关参数（渔网宽度、清鱼是否需要网完全展开、补鱼速率等）进 `config.js` 可调。

**阶段 3 自检**：渔网下落经过鱼群时，带内的鱼被捕消失、带外的鱼不受影响；被捕不卡顿（粒子受控）；渔网清场后世界鱼群能逐步补回、不会越玩越空；封神巡游时能体验到"追鱼群、躲渔网、被迫转移"的节奏。
