# 文字地牢（文字版暗黑 / 刷子 Roguelike）· 设计稿 v0.1

> 合集第二款游戏。位于 `dungeon/` 子目录，独立运行、独立存档。
> 定位：**重玩法（数值养成）、极简表现**。以文字/数据为骨，配极简像素图标 + 血条。
> 技术栈延续合集：纯前端 · 原生 HTML/CSS/JS · ES Modules · 零依赖 · Canvas 只用于极小图标（可选，甚至纯 DOM 即可）。
> 第一版进度存 **localStorage**；后期迁到合集统一 SQLite 后端（见文末）。

---

## 一、一句话定位

选一个职业，配好装备与加点，**下地牢自动战斗**，掉落更好的装备，穿上后挑战更深的层数。
核心爽点 = **「刷 → 变强 → 刷更深 → 掉更好的装备」的循环**（暗黑的灵魂）。
核心张力 = **风险回报的抉择**（要不要赌一把打更深的层，死了这趟白刷 vs 稳妥刷已通层数）。

---

## 二、核心循环（Core Loop）

```
【角色面板】职业 / 等级 / 属性 / 装备栏 / 背包
      │
      ▼
【选择地牢层数】第 1 层 … 已解锁的最深层（越深越难、掉落越好）
      │  点「进入」
      ▼
【自动战斗】一波怪，双方按公式自动结算，战斗日志逐行滚动
   "🔥火球命中骷髅，造成 127 伤害"  "骷髅反击，你 -40 HP"  …
      │  一方倒下
      ▼
【结算】胜：获得经验 + 金币 + 掉落装备（白/蓝/紫/橙，随机词条）
        败：掉一部分经验/金币（死亡惩罚方案 B，有痛感但不删角色）
      │
      ▼
【养成】穿更好的装备 / 升级加点 / 学技能 → 挑战更深的层 → 回到循环
```

一趟战斗 30 秒内看完，符合「随时玩一把」的休闲节奏，又有长期养成的深度。

## 三、驱动方式：自动战斗 + 战前配装（已定）

- **玩家不手动点每一次攻击。** 进入地牢后，战斗**自动结算**：双方按属性/技能轮流行动，玩家看战斗日志滚动 + 血条变化。
- **玩家的操作集中在「战前」**：选职业、加点、穿装备、配技能、选打第几层。**深度在配装（build），不在手速。**
- 战斗表现：**日志逐行滚动**（可加轻微打字机/淡入效果）+ 双方血条 + **emoji 图标**（角色/怪物/技能用 emoji，见第九节）。掉落时装备卡片弹出。
- **日志偏长、有细节**（命中/暴击/闪避/技能触发/阵亡/掉落都播报），让战斗有"戏"；**但必须配加速与跳过**（已定）：
  - **加速档位：1× / 2× / 3×**（切换日志吐出速度）。
  - **跳过**：直接结算，跳到战斗结果（给刷子后期省时间）。
- 可选的少量「战中交互」（第二版再考虑，第一版先纯自动）：紧急喝药、放一个大招。第一版**保持纯自动**，把核心循环跑通。

## 四、战斗系统：群体战斗（1 名玩家 vs 一小群怪）

**已定：群体战斗**（像传统暗黑，你同时面对一群怪）。这是战术深度的来源，但会引入大量边界情况，**第七节专门兜底，GPT 必须逐条处理**。

### 4.1 基本结构
- 一波战斗：**玩家 1 人 vs 敌人一群（第一版建议 3~5 只，数量进 config）**。
- **简单回合制自动结算**：所有存活单位按「速度」从高到低排序，依次出手，一轮结束再来一轮，直到一方全灭。
- 战斗全自动，玩家不输入；中途只能「撤退」（触发死亡惩罚的减档版 or 保留部分收益，见第七节）或看完/加速/跳过。

### 4.2 单次伤害公式（示意，数值进 config 可调）
`伤害 = (攻击力 × 技能倍率 - 目标防御 × 减免系数) × 暴击修正 × 随机浮动`
- 暴击：按暴击率触发，暴击伤害倍率（如 ×1.5）。
- 随机浮动：±10% 左右，让每次不完全一样（用可复现种子 RNG）。
- **保底**：伤害有下限（≥1，不会被防御减到 0 或负），避免卡死。

### 4.3 一层的构成
- 第一版：**一层 = 一波怪**；**每 5 层的那层是 Boss 层**（Boss 单体、血厚、伤害高，可带机制）。
- 后续可扩展为「一层多波」。

## 五、职业系统（第一版先做 1 个，架构预留多职业）

**职业区别的本质 = 数值 + 技能 + 适配词条，不是外观。** 文字/数据版反而最容易做出区别。

规划 3 个职业（第一版只做**战士**跑通，其余留接口）：

| 职业 | 主属性 | 特点 | 代表技能 |
|------|--------|------|----------|
| 战士 Warrior | 力量 | 高血高防、近战物理、耐揍 | 重击、格挡、旋风斩 |
| 法师 Mage | 智力 | 低血高爆发、魔法伤害、有法力资源 | 火球、冰冻、连锁闪电 |
| 游侠 Ranger | 敏捷 | 中等、高暴击、多段攻击、高闪避 | 多重射击、穿透箭、疾步 |

- **属性系统（3 主属性）**：力量（物理攻）、敏捷（暴击/闪避/速度）、智力（魔法攻/法力）。外加**体质→生命值**。
- 每个职业主属性不同、伤害类型不同、成长曲线不同、**吃的装备词条不同**（战士堆力量/生命，法师堆智力/暴伤，游侠堆敏捷/暴击）——区别自然产生。

### 5.1 加点：两种模式（已定）
- **自动推荐加点**（勾选）：按职业推荐权重自动分配属性点，休闲省心。
- **手动加点**：玩家自由分配，追求 build 深度。
- 两模式可切换；升级获得属性点，未分配的点保留。

### 5.2 技能分三类（已定：区分单体/群体）
群体战斗下，技能类型带来真正的战术意义：
- **单体技能**：高倍率，集中打一个目标 → 用来秒高威胁怪 / 打 Boss。
- **群体技能**：范围伤害，打全部敌人，但**单目标伤害明显低于单体技能**（用覆盖面换单点效率）→ 怪多时高效清场。
- **增益/自保技能**：减伤、拉仇恨、回血、加攻等，不直接造成伤害。

战士第一版技能示例：
| 技能 | 类型 | 效果 | 冷却 |
|------|------|------|------|
| 普通攻击 | 单体 | 基础伤害，无冷却（兜底技能） | 0 |
| 重击 | 单体 | 高倍率打一个 | 2~3 回合 |
| 旋风斩 | 群体 | 打全部敌人，单体伤害较低 | 3~4 回合 |
| 格挡 | 增益 | 数回合内大幅减伤 | 4~5 回合 |

> **技能要有冷却**，否则自动战斗只会无脑放最强技能，技能选择失去意义。冷却让 AI 施法有节奏、有取舍。

### 5.3 自动战斗的施法 AI（关键，决定战斗好不好看）
玩家不手动放技能，所以**需要一套简单的施法决策逻辑**，让战斗有策略感而非乱放：
- 敌人 ≥ 阈值（如 3 个）且群体技能可用 → 优先放群体技能。
- 只剩 1~2 个敌人 → 用单体技能集中打（别浪费群攻）。
- 自身血量低于阈值 → 优先放自保/回血（若有）。
- 都在冷却 → 用普通攻击兜底。
- 决策规则做成**可配置的优先级表**，进 config，方便调。

## 六、群体战斗兜底清单（⚠️ GPT 必须逐条处理，否则必出 bug）

群体战斗（多个单位互相攻击）会产生大量边界情况。以下每一条都要在战斗模拟器里明确处理，不能含糊：

### 7.1 目标选择
- **玩家单体技能打谁**：默认策略「优先打当前血量最低的敌人」（更快清场、减少挨打）；做成可配置策略（最低血 / 最高攻 / 最近），进 config。
- **敌人打谁**：第一版只有玩家一个可攻击目标，所有敌人都打玩家（简单）。架构上预留「多目标」的可能，但第一版不做。
- **目标在技能结算前已死**：若一个技能锁定的目标在轮到它前已阵亡 → 自动重新选一个存活目标；无存活目标则该次行动跳过（不报错）。

### 7.2 出手顺序 & 回合
- 每轮开始，对**所有存活单位**按速度从高到低排序出手。
- **速度相同的 tie-break**：用稳定的次序（如按 id / 初始位置），**不能用随机**，否则同种子结果不可复现、不可测试。
- **回合上限兜底**：设一个最大回合数（进 config，如 200）。超过仍未分出胜负 → 判定为「玩家撤退/失败」并结束，**绝不允许死循环**。
- 每次出手前重新检查该单位是否还活着（可能已被本轮前面的单位打死）→ 死了就跳过。

### 7.3 群攻 vs 单体的平衡
- **群攻的单目标伤害必须低于单体技能**，用「打多个」换「单点弱」，否则群攻碾压单体、单体技能没用。
- 施法 AI 要体现取舍（见 5.3）：怪多放群攻、怪少放单体。

### 7.4 技能资源 / 冷却
- 每个主动技能有冷却回合数，出手时递减；冷却中不可用。
- 玩家**至少有一个 0 冷却的普通攻击兜底**，保证任何回合都有得打。
- （若引入法力/怒气资源）资源不足时该技能不可用，回退到兜底攻击。

### 7.5 死亡 / 清场时机
- 一次群攻同时打死多只怪 → 每只的掉落/经验/金币**都结算**，合并展示。
- 玩家在自己出手前被打死 → 立即判负，不再出手，进入死亡惩罚结算。
- 敌人全灭的瞬间战斗立即结束，本轮尚未行动的敌人不再出手。
- 掉落结算在战斗**完全结束后**统一进行，避免边打边弹。

### 7.6 战斗日志可读性（群战日志容易刷屏）
- 每条日志标明「谁 对 谁 做了什么、数值多少」。
- **群体技能合并成一条**（如「旋风斩命中 3 个敌人，共 X 伤害」）或归组显示，不要刷出十几行。
- 暴击、闪避、技能触发、单位阵亡、掉落，都要有清晰可读的一行。
- 配合加速（2×/3×）和跳过（直接出结果）——长日志好看，但必须能快进。

### 7.7 空 / 异常状态兜底
- 一波怪数量为 0（不该发生）→ 直接判胜，不进战斗循环。
- 玩家没有任何主动技能 → 用普通攻击。
- 所有数值 clamp：血量不为负、伤害 ≥ 1、防御减免不超过某上限（避免无敌）。
- 任何 NaN/undefined 数值都要有默认值兜底，单个异常不能让整场战斗崩溃。

> 战斗模拟器写成**纯函数**（输入双方初始状态 + 种子 → 输出完整战斗日志 + 结果），这样以上每一条兜底都能写成**单元测试**逐一验证。这是第一版最该认真测的部分。

## 七、装备与词条系统（刷子的灵魂 —— 最重要）

这是「多巴胺来源」，也是最需要认真设计、最容易失控的部分。**第一版必须极度克制。**

### 6.1 装备部位（第一版 4 个）
武器 / 头盔 / 护甲 / 饰品。（第二版再加：手套、鞋、副手、戒指等）

### 6.2 稀有度（4 档）
| 稀有度 | 颜色 | 词条数 | 掉率（示意） |
|--------|------|--------|------|
| 普通 | 白 | 0~1 | 高 |
| 优秀 | 蓝 | 1~2 | 中 |
| 稀有 | 紫 | 2~3 | 低 |
| 传说 | 橙 | 3~4 + 特效 | 很低 |

稀有度越高：基础数值越高 + 词条越多 + 越可能带特殊词条。**橙装可带一个独特特效**（如"攻击附带燃烧""暴击回血"），是玩家追逐的终极目标。

### 6.3 词条（随机生成，第一版一小池）
装备生成时，从词条池按稀有度随机 roll 出若干条，每条数值也在范围内随机：
- 基础词条：+力量 / +敏捷 / +智力 / +生命 / +攻击 / +防御
- 进阶词条（紫橙才有）：+暴击率 / +暴击伤害 / +攻速 / +伤害% / +减伤%
- 特效词条（橙专属，第一版可先 3~5 种）：燃烧、吸血、反弹、破甲、连击

> **同一件装备的随机性**，就是让玩家「再刷一件说不定更好」的动力。同名武器可能因为词条不同，一件废一件神。

### 6.4 装备比较 & 替换
掉落时清楚展示「新装备 vs 当前装备」的属性差异（↑绿 ↓红），玩家一眼判断要不要换。可加「自动标记更优」提示。

## 八、成长与难度曲线（第二个易失控点）

- **层数递增**：越深的层，怪物血量/攻击/防御越高，但**经验/金币/掉落品质也越高**。
- **曲线目标**：让玩家「刚好打得过当前解锁的最深层，再深一点就有点吃力」——始终有个够得着的目标。避免「太肝（升级太慢）」和「无脑（一路平推）」两个极端。
- **解锁机制**：通过第 N 层才能挑战第 N+1 层；已通过的层可重复刷（刷装备/经验）。
- **数值全部进 config**，做成可调曲线（怪物成长率、掉落品质随层提升的曲线、经验需求曲线等），方便反复调平衡。

> ⚠️ **数值黑洞警告**：刷子游戏极易越加越复杂、平衡永远调不完。第一版死守极简：
> **1 职业、3 属性、4 部位、4 稀有度、一小池词条、10~15 层**。先把「刷→变强→刷」循环跑通、确认好玩，再扩。

## 九、界面（极简，文字为主）

- **主面板**：角色信息（职业/等级/经验条/属性）、装备栏（4 部位 + 图标）、背包、金币。
- **地牢选择**：一列层数按钮，标注难度/推荐战力/已通过状态。
- **战斗界面**：双方名字 + 血条 + 极简像素头像；中间/下方是**滚动战斗日志**；结束弹结算。
- **掉落/背包**：装备用卡片列表，颜色区分稀有度，hover/点击看词条明细与对比。
- 视觉：延续合集调性但更偏「暗黑地牢」的暗色。**图标一律用 emoji**（已定）——武器 ⚔️🗡️🏹、防具 🛡️⛑️、职业/怪物 💀👹🐗👑、稀有度/掉落 💎🔥、技能 💥🌀 等。生动、跨平台一致、零绘制成本。**不做实时移动地图。**

## 十、技术要点

- **纯数据驱动**：角色、装备、怪物、词条都是 JS 对象/数组。战斗是一个纯函数式的模拟器（输入双方状态 → 输出战斗日志 + 结果），**天然好测试**（可像大鱼那样写纯逻辑单元测试）。
- **随机数用可复现的种子 RNG**（复用 bigfish 里的 `createSeededRng`/`hashSeed` 思路），方便调试和潜在的「固定种子地牢」玩法。
- **无实时循环压力**：不像大鱼要 60 帧渲染。战斗日志可以定时逐行吐出（`setTimeout`/`requestAnimationFrame` 控制节奏），CPU 压力极小。
- **存档（第一版）**：localStorage。存角色、装备、背包、已解锁层数、金币。结构清晰、字段命名规范，**为将来迁移到 SQLite 后端预留**（一份角色存档 = 一个 JSON，将来直接往后端塞）。
- **模块建议**：`combat.js`（战斗模拟器）/ `loot.js`（装备词条生成）/ `hero.js`（角色/职业/属性）/ `dungeon.js`（层数/怪物）/ `save.js`（存档）/ `ui.js`（渲染）/ `game.js`（串联）。分模块 + 每块可测。

## 十一、后端时机（对齐合集规划）

- **第一版：localStorage 单机**。先把核心循环做好玩，好不好玩与后端无关，别为存档提前上后端。
- **好玩之后：迁到合集统一后端（SQLite）**。刷子游戏是长期养成，存档必须安全 + 跨设备，这是硬需求——但**等游戏本身成立了再上**。
- 统一后端管：账户、跨游戏存档、排行榜。SQLite 一个文件足矣（十万玩家几个 GB），VPS 无压力。届时把 localStorage 存档结构原样往后端搬即可（所以第一版存档结构就要设计干净）。

## 十二、开发路线（分步，每步可玩/可验证）

1. **战斗模拟器内核**：给定「战士 vs 一只怪」，输出战斗日志和胜负。纯逻辑，先写测试，无界面也能验证。
2. **最小可玩循环**：一个固定角色 → 打一层固定的怪 → 赢了给一件固定装备 → 穿上变强 → 再打。跑通「刷→变强」。
3. **装备词条随机生成**：接入 loot 系统，掉落随机稀有度/词条的装备 + 背包 + 穿脱 + 属性对比。
4. **层数与难度曲线**：多层地牢、怪物随层成长、解锁机制、重复刷。
5. **升级与加点**：经验/等级/属性成长。
6. **技能系统**：给战士几个技能，战斗中按逻辑触发。
7. **多职业**：加法师、游侠，验证职业差异。
8. **打磨**：UI、掉落反馈、音效（可复用/程序化）、平衡调校。
9. **（后期）接入合集统一后端**：账户 + 云存档 + 排行榜。

> 每一步都是「能验证的东西」，做完一步再叠下一步。第 1~2 步就能判断「这个循环好不好玩」——这是最关键的早期验证点。
